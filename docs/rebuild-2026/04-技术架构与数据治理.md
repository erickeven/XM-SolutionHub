# 04 技术架构与数据治理

## 1. 架构结论

新系统采用**模块化单体 + MySQL 9.7 LTS + Worker + Adapter**。这是单台局域网服务器和开发阶段的最小正确架构：业务规则在一个可测试、可事务化的服务中，慢任务和外部集成异步化，搜索/文件转换/身份认证预留可替换边界。

不采用微服务、Kafka、Kubernetes、独立搜索集群或门户 SSR 作为第一阶段的强制部署项。它们仅在可量化需求出现后启用。

## 2. 第一阶段部署拓扑

```text
Nginx -> Client static site -> API -> MySQL 9.7 LTS
                                  -> Redis
                                  -> SeaweedFS 4.29 (S3)
                                  -> Worker
```

| 组件 | 第一阶段责任 | 不承担 |
| --- | --- | --- |
| Client | 外部发现、个人/客户协作、内部作业台 | 直接访问数据库或对象存储 |
| API | 鉴权、业务规则、策略、读写、短时文件授权 | Office 转换、长时间索引、慢速外部调用 |
| Worker | 预览生成、压缩包、通知、索引、集成重试 | 绕过业务权限直接发布事实 |
| MySQL | 产品、方案、文件元数据、授权、项目、工作流、审计、Outbox 的唯一事实源 | 大规模向量 ANN 索引 |
| Redis | 缓存、限流、可丢失短任务 | 不可恢复业务事实 |
| SeaweedFS | 通过 S3 Adapter 保存原件、预览派生物、压缩包和对象版本 | 权限判断和业务版本关系 |

Docker Compose 采用开发/生产覆盖文件。生产服务使用固定镜像版本，新数据库初始化、管理员初始化和服务启动分离；服务器上不得在启动时临时下载依赖。

## 3. 技术基线

| 层级 | 最终选择 | 约束 |
| --- | --- | --- |
| 前端 | React 19 + TypeScript + Vite | 新代码库直接采用 React 19；UI 由共享设计令牌约束 |
| 后端 | Node.js 24 LTS + TypeScript + Express | 新代码库强制 Route -> Controller -> Service -> Repository -> Adapter 分层 |
| ORM | Prisma 当前受支持稳定版 | 新代码库从 MySQL 9.7 schema 开始；关键 DDL/全文索引/复杂查询允许封装原生 SQL |
| 事务库 | MySQL 9.7 LTS | 固定到验证过的补丁版本或镜像摘要；InnoDB、UTC、`utf8mb4`、binlog 与恢复演练为上线前置 |
| 缓存/异步 | Redis + 独立 Worker + MySQL Outbox | 发布、授权、同步等关键事件以 Outbox 为准，必须幂等、可重放 |
| 对象存储 | SeaweedFS 4.29 / S3 Adapter | bucket 版本化、生命周期、短时授权、按版本审计；业务代码不得依赖厂商专有接口 |
| 测试 | Vitest、Supertest、RTL、Playwright | 选型算法、授权、字段配置、文件预览、关键客户旅程必须自动化 |

Node 24 是当前 LTS，新代码库以其作为生产运行时基线。[Node.js releases](https://nodejs.org/en/about/previous-releases)

正式依赖锁定时采用 TypeScript 6.0.3。虽然 TypeScript 7.0.2 已发布，但当前 `typescript-eslint` 8.63.0 的声明支持范围为 TypeScript `>=4.8.4 <6.1.0`；在其正式支持 TypeScript 7 前不得升级，以保证 lint、类型检查和 CI 使用受支持组合。

## 4. 逻辑模块与适配边界

```text
Route -> Controller/Schema -> Application Service -> Domain Policy
      -> Repository(MySQL) / Adapter(Storage, Search, CRM, ERP, Mail)
      -> Outbox -> Worker
```

| 模块 | 拥有的事实 | 必须隔离的外部能力 |
| --- | --- | --- |
| IAM | 身份、组织、角色、属性、NDA、授权策略 | SSO/企业身份认证适配器 |
| PIM | 产品、料号、参数、封装、认证、替代 | ERP/PDM 数据适配器 |
| Solution Asset | 方案、BOM、测试、文件、发布包、适配关系 | 对象存储、文档转换适配器 |
| Selection | 需求、约束、候选、比较快照、解释 | 搜索与规则适配器 |
| Project/Support | 企业、项目、工单、成员、时间线 | CRM、样品、邮件适配器 |
| Supply/Quality | 生命周期、PCN、合规、样品、RFQ、质量事件 | ERP、渠道、实验室适配器 |
| Content/Config | 页面、文案、字段、菜单、配置版本 | 发布/预览适配器 |
| Analytics/Knowledge | 事件、受控知识、指标和 AI 反馈 | 搜索/模型/BI 适配器 |

## 5. MySQL、检索与全新初始化

MySQL 9.7 是业务事务库，不是所有检索问题的默认答案。其 `VECTOR` 列不能属于任何 key，不能用作大规模 ANN 向量索引；系统必须使用 `SearchAdapter` 隔离检索实现。[MySQL VECTOR](https://dev.mysql.com/doc/refman/9.7/en/vector.html)

| 阶段 | 启用能力 | 启用条件 |
| --- | --- | --- |
| 默认 | MySQL `ngram` 中文全文、结构化筛选、关系扩展、权限过滤 | 第一阶段必做；CJK 全文索引必须使用 `WITH PARSER ngram`，默认解析器不得作为中文检索实现 |
| 可选搜索服务 | 同义词、拼写纠错、复杂聚合和混合检索 | 真实资料集证明 MySQL 检索的零结果率、P95 或召回不足 |
| 可选向量索引 | 受权限控制的语义召回和知识问答 | 评测集证明提升召回且来源正确率、权限拒答和成本达标 |

Prisma 支持 MySQL 连接器，但这不替代 MySQL 9.7、中文全文、schema DDL、查询计划和备份恢复 POC。[Prisma MySQL connector](https://www.prisma.io/docs/orm/core-concepts/supported-databases/mysql)

中文全文索引由受控原生迁移创建，采用 `WITH PARSER ngram`，第一阶段固定 `ngram_token_size=2` 并用真实术语评测集校验召回。Prisma schema 负责表、关系和普通索引，不得用其默认 `@@fulltext` 代替 CJK 解析器配置；检索查询统一封装在 `SearchAdapter`。MySQL 默认全文解析器无法识别无空格的中文词边界，InnoDB 全文索引的事务内新增内容仅在提交后可检索。[MySQL ngram parser](https://dev.mysql.com/doc/refman/9.7/en/fulltext-search-ngram.html)

新系统采用最终 MySQL schema 干净初始化，只创建管理员、角色、权限和必要系统配置。现有项目的 PostgreSQL schema、迁移记录、`pgvector` 代码、演示数据和文件均不进入新代码库或新部署环境；本阶段不存在任何数据迁移或兼容层。

## 6. 文件、搜索与 AI

- 原件、预览 PDF、页图、文本抽取和压缩包有独立对象键、输入版本与生成状态；原件不能就地覆盖。
- 第一阶段优先稳定 PDF 预览。若 Office 资料预览的真实需求和量级成立，再由 Worker 接入独立转换容器；不要把转换放进 API 请求。
- 阶段 0 必须用真实 PDF 验证页数、前三页派生物、版本键和失败处理。若第一阶段真实样本包含 Office 原件，则 Office 到 PDF 转换也必须通过；若不包含，则先验证 `DocumentConversionAdapter` 的排队、幂等、超时、失败可见和指标采集，不能为了勾选清单伪造转换成功或把 LibreOffice 装进 API 镜像。
- 可选转换服务可采用 Gotenberg 一类 Docker API，负责 Office 到 PDF；中文字体、复杂 Excel/Word 分页和失败重试必须纳入 POC。[Gotenberg](https://gotenberg.dev/docs/convert-with-libreoffice/convert-to-pdf)
- AI 只检索当前用户有权访问、已发布且未失效的版本；输出必须携带来源与未知项，不能写回产品/方案/质量事实或自动发布。

## 7. 安全、审计与可运维性

| 领域 | 最终要求 |
| --- | --- |
| 文件访问 | 短时授权、服务端策略二次校验、下载审计、异常限流；不返回永久对象地址 |
| 审计 | 记录读取受限内容、下载、导出、授权、发布、删除、审批和管理员覆盖的对象版本与前后值 |
| 备份恢复 | MySQL 全量备份 + binlog 时间点恢复、SeaweedFS 数据卷与 S3 对象版本恢复、季度恢复演练 |
| 可观测性 | 统一 trace ID、结构化日志、请求/队列/预览/搜索/授权指标；先接入基础追踪，按规模再扩展完整平台 |
| 密钥 | 生产密钥只在部署环境注入，不进入仓库、镜像或日志 |

对象存储选型于 2026-07-13 触发阶段 0 风险 R-001：MinIO 社区仓库已归档，最后可用容器镜像经 Trivy 0.72.0 检出 71 项可修复 HIGH/CRITICAL 漏洞，因此不得进入生产。经架构/安全评审改用仍受维护、支持 S3 API 与对象版本化的 SeaweedFS 4.29；应用只依赖 `ObjectStorageAdapter`，不引入迁移或兼容旧存储的代码。[SeaweedFS releases](https://github.com/seaweedfs/seaweedfs/releases) [SeaweedFS S3 quick start](https://github.com/seaweedfs/seaweedfs#quick-start-with-weed-mini)

Bucket 版本化是对象锁与保留策略的前置能力；是否启用对象锁由资料/审计保留要求决定，不能对临时预览无差别使用。生产上线前必须重新通过对象版本恢复、短时授权、原件/派生物读写和镜像扫描。
