# 00 总体决策与范围

> 状态：最终定型，作为下一阶段全新重写的业务与技术基线。  
> 适用对象：芯茂微管理层、业务负责人、工程/质量/AE/客户管理部门、研发与测试团队。  
> 不适用：本文件不替代接口文档、全新初始化脚本、部署手册或具体页面原型。

## 1. 平台定位

芯茂微数字化 Design-in 平台服务两类主体：

1. 客户以最低阻力完成找型号、找方案、找资料、选型、验证和采购协作。
2. 芯茂微以受控数据和跨部门协同推动 Design-in、样品/询价、质量支持、验证和量产经营。

平台不是普通官网、资料下载站、CRM、ERP 或在线商城的替代品。它是产品/方案事实、受控资料和客户技术协同的中枢；CRM、ERP、样品和渠道系统通过适配层交换各自负责的事实。

## 2. 不可改变的业务主链路

```text
客户进入 -> 找型号 / 找方案 / 找资料 / 快速选型
-> 系统推荐适配产品 -> 产品详情
-> 明显展示关联应用方案 -> 方案详情
-> 预览方案资料 -> 按授权完整预览、下载单份或全包
-> 仅在客户主动需要协作、样品、RFQ 或专项支持时进入企业/项目流程
```

以下规则是后续设计和代码的硬约束：

- 热门应用展示和筛选对象是**方案**；推荐型号展示和筛选对象是**产品/可订货料号**。
- 产品详情必须突出可用方案，方案详情必须突出版本、适用条件、测试证据和资料完整度。
- 公开资料允许匿名预览前三页；注册个人按资源分级获得完整预览或下载。不能以注册、企业认证或开项目作为“找方案/找资料”的前置条件。
- 产品字段和资料字段由字段定义管理，至少支持文本、数字、布尔及业务需要的枚举/日期/文件类型；字段类型不是写死的页面逻辑。
- 所有文件访问使用短时授权，原始存储地址不得直接暴露。

## 3. 客户分层与项目边界

| 层级 | 客户意图 | 允许能力 | 不应发生的事 |
| --- | --- | --- |
| L0 匿名 | 查资料、找方案、初步选型 | 搜索、选型、公开内容前三页预览 | 被强迫注册、建企业或开项目 |
| L1 注册个人 | 保存、收藏、订阅、下载开放资料 | 个人工作台、完整预览/下载开放资料、通用咨询 | 自动成为企业成员或自动创建商机 |
| L2 企业协作 | 共享资料和团队协作 | 企业成员、企业授权、共享选型和订阅 | 因加入企业而被强制开正式项目 |
| L3 正式项目 | 专项技术支持、样品、RFQ、Design-in | 需求版本、候选、设计包、样品/RFQ、受控协同 | 用一次浏览或下载伪造项目事实 |

客户 FAE/工程师、采购、PM 是默认任务视图和协作职责，不是公开浏览或个人资料使用的权限门槛。

## 4. 芯茂微应获得的价值

| 目标 | 平台提供的能力 | 可观察结果 |
| --- | --- | --- |
| 技术获客 | 可信产品参数、方案与资料发现 | 找方案/资料成功率、收藏和咨询转化 |
| Design-in 推进 | 项目需求、候选、证据、FAE 支持、测试验证 | 技术首响、结案、方案复用、项目阶段转化 |
| 供应连续与客户服务 | 生命周期、PCN、样品、RFQ、跟单交接 | 风险确认、样品时效、采购资料完整度 |
| 质量与技术可信 | 版本、审批、测试原始证据、发布与回滚 | 资料版本覆盖、过期率、质量处理时效 |
| 经营决策 | 可追溯漏斗、数据质量和资源负载 | 有效 Design-in、量产转化、部门 SLA |

## 5. 可配置的边界

系统管理员可以管理全部前端页面、导航、文案、字段定义、字段分组、展示顺序、资源分级、权限策略和流程配置。配置必须版本化、可预览、可发布、可回滚并留下审计记录。

“可配置”不等于删除业务约束：产品/料号/方案/文件的关系、审批状态、授权规则、审计事件和数据校验仍由领域模型保证。管理员可全局覆盖，但覆盖动作必须明确标识并留痕。

## 6. 全新重写边界、首阶段范围与明确不做

### 全新重写边界

现有项目的源代码、数据库 schema、迁移记录、演示数据、对象存储文件、环境配置、Docker 镜像、部署目录和运行实例均不是新系统资产。新系统从空仓库结构、空 MySQL、空对象存储和全新 Docker Compose 创建；仅本资料包中重新确认的业务目标、角色和验收规则可作为设计输入。

### 首阶段必须交付

- 产品、可订货料号、参数字段、方案、文件、版本、发布和授权。
- 公开发现、个人工作台、产品/方案/资料详情、前三页预览、完整预览/下载。
- 芯茂微销售、普通 FAE、样品/跟单、质量、AE、管理者和系统管理员的最小作业能力。
- MySQL 9.7、文件对象存储、审计、备份恢复、Docker Compose 和自动化测试基线。

### 当前明确不做

- 不把 CRM、ERP、商城、实时库存、实时价格、CAD 在线编辑或自动设计结论一次纳入。
- 不为了“企业化”先拆微服务、Kafka 或 Kubernetes。
- 不把 AI 输出当作无来源的技术结论；AI 只能基于当前用户有权访问的受控证据工作。
- 不迁移或复用现有项目的任意数据、代码、配置、文件或部署产物；新环境仅初始化管理员、角色、权限和必要系统配置。

## 7. 开发开始的唯一门槛

只有 [05-实施路线图与验收](05-实施路线图与验收.md) 的阶段 0 POC 与 [06-立项与验证清单](06-立项与验证清单.md) 的签字项完成，才开始阶段 1 及后续正式交付代码的开发。阶段 0 允许创建隔离、可重复运行、可整体丢弃且不进入生产部署的验证代码；该代码只用于形成 POC 证据，不得在未经评审的情况下直接视为正式实现。
